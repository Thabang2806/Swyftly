# Swyftly Database Schema

The database is PostgreSQL with EF Core migrations.

## Current State

The scaffold includes a `SwyftlyDbContext` and default schema name `mabuntle`.

EF Core migrations live in `backend/src/Swyftly.Infrastructure/Persistence/Migrations` and are generated through the Infrastructure project with the API project as startup.

Current migrations create:

- ASP.NET Core Identity tables for users, roles, claims, logins, and tokens.
- `buyer_profiles`, linked one-to-one to Identity users, with optional lightweight buyer display name and phone number settings.
- `buyer_notification_preferences`, linked to buyer profiles, storing category-level in-app and email notification opt-in state for Orders, Returns, Reviews, and Support. Missing rows default both channels to enabled behavior.
- `buyer_delivery_addresses`, linked to buyer profiles, storing reusable checkout delivery addresses with one default address per buyer when addresses exist, optional delivery instructions, and local address verification status/provider/warnings metadata.
- `buyer_wishlist_items`, linked to buyer profiles and products, enforcing one wishlist item per buyer/product.
- `buyer_growth_events`, linked to buyer profiles and optionally products, storing sanitized AI discovery telemetry for `/assistant` and `/visual-search`: event type, source tool, result count, confidence band, derived category/colour/material context, source route, optional structured feedback reason, and occurrence timestamp. The table intentionally does not store raw prompts, uploaded image data, previews, base64 content, provider payloads, full AI responses, or free-text feedback.
- `buyer_growth_outcomes`, linked to buyer profiles and optionally buyer growth events, products, carts, and orders, storing privacy-conscious AI discovery attribution outcomes for `ProductOpened`, `ProductAddedToCart`, `CheckoutStarted`, `OrderCreated`, and `OrderPaid`. Rows store source tool, optional confidence band, occurrence time, attributed-from timestamp, and attribution-window minutes. Phase 11J uses a 7-day window and records outcomes best-effort after existing buyer/cart/order/payment workflows succeed. The table does not store prompts, uploaded image data, previews, base64 content, provider payloads, full AI responses, free-text feedback, payment-provider internals, or buyer-facing personalization data.
- `buyer_ai_discovery_preferences`, linked one-to-one with buyer profiles, stores opt-in AI history and opt-in AI personalization flags. Both default disabled when no preference row exists. Phase 11L adds `PersonalizationEnabled`; personalization reads only safe buyer-owned shopping/history summaries and does not store raw prompts, uploaded image data, provider payloads, support/dispute text, payment/refund data, or free-text feedback.
- `buyer_ai_discovery_preferences`, linked one-to-one to buyer profiles, storing the opt-in server-side AI discovery history preference. Missing rows default history to disabled.
- `buyer_ai_discovery_history`, linked to buyer profiles, storing opt-in safe summaries for assistant and visual-search results: source tool, derived category/colour/material, confidence band, result count, returned product ids, optional source route, and creation timestamp. This table intentionally does not store raw prompts, uploaded image data, previews, base64 content, provider payloads, full AI responses, or free-text feedback.
- `seller_profiles`, linked one-to-one to Identity users, with seller profile fields and `PendingVerification` as the initial seller status.
- `seller_notification_preferences`, linked to seller profiles, storing category-level in-app and email notification opt-in state for Verification, Products, Revisions, Ads, and Reports. Missing rows default both channels to enabled behavior.
- `seller_report_schedules` and `seller_report_schedule_runs`, linked to seller profiles, storing opt-in weekly/monthly analytics digest settings, next/last run metadata, failure metadata, and idempotent scheduled report run records. Reports are read-only digest delivery metadata and do not mutate marketplace operational records.
- `seller_funnel_events`, linked to sellers and optionally products, carts, orders, buyers, storing first-party storefront conversion events (`StorefrontViewed`, `ProductViewed`, `ProductAddedToCart`, `CheckoutStarted`, `OrderCreated`, `OrderPaid`) with hashed anonymous visitor id, source route, optional UTM source/medium/campaign, referrer host, source category, occurrence timestamp, and optional idempotency key. These rows power seller funnel/source reporting and never mutate checkout, payment, payout, carrier, or inventory state.
- `seller_storefronts`, linked one-to-one to sellers, with a unique storefront slug.
- `seller_addresses`, linked one-to-one to sellers for onboarding and fulfilment location data.
- `seller_store_policies`, linked one-to-one to sellers, storing buyer-facing informational return, exchange, fulfilment, support, care, and product disclaimer policy text plus an optional return window. Updates are seller-owned and audited; policies provide context only and do not automate return/refund decisions.
- `seller_delivery_methods`, linked to sellers, storing provider-free checkout delivery rates with Standard/Express/LocalCourier/PickupPoint method type, country/province coverage, base price, optional free-shipping threshold, delivery-day estimate, display order, active flag, and audit timestamps.
- `pickup_points`, platform-managed pickup locations with provider/code identity, address, optional coordinates/opening hours, active flag, and audit timestamps. Checkout matches active pickup points by country and province for seller `PickupPoint` delivery methods.
- `seller_payout_profiles`, linked one-to-one to sellers, storing payout provider references/placeholders only, not sensitive bank data.
- `seller_payout_profile_change_requests`, linked to sellers, storing verified-seller payout provider reference change requests with Draft/PendingReview/Approved/Rejected/Cancelled status, requester/reviewer ids, review reasons, timestamps, and a concurrency token. A filtered unique index allows only one active draft or pending request per seller.
- `seller_verifications`, linked many-to-one to sellers for verification review history.
- `seller_verification_evidence`, linked to sellers, storing private seller verification evidence metadata only: evidence type, private storage provider/key, original file name, content type, byte size, SHA-256 hash, optional note, upload actor/timestamp, and soft-remove actor/timestamp. Evidence files are served only through authorized API download endpoints and are not exposed on public catalog/storefront APIs.
- `refresh_tokens`, linked to Identity users and storing hashed refresh tokens only, token-family ids for replay revocation, replacement hashes, and revocation reasons.
- `audit_logs`, currently used for admin seller approval/rejection/suspension actions, seller verification evidence upload/remove/admin-download actions, seller payout-profile change submission/review actions, admin product approval/rejection/change-request actions, admin review moderation actions, seller inventory adjustments and bulk stocktakes, admin historical inventory-ledger backfills, seller return restock decisions, seller delivery-method create/update/activate/deactivate actions, admin pickup-point create/update/activate/deactivate actions, and admin category/attribute create/update/activate/deactivate actions.
- `categories`, with parent-child hierarchy, unique slug, display order, and active status. Admin write operations use deactivation rather than hard delete.
- `category_attributes`, linked to categories, with data type, required flag, display order, active status, and JSONB allowed values for select-style attributes. Admin write guards prevent breaking edits where existing product attribute values depend on a key, type, required flag, or allowed value.
- `products`, linked to sellers and optionally categories/brands, with seller-scoped slug uniqueness, draft/review/publication statuses including `NeedsAdminReview` and `ChangesRequested`, rejection/change-request reason, publish timestamp, JSONB tags, and seller-managed SEO/merchandising/care/disclaimer presentation fields.
- `product_variants`, linked to products, with SKU, size, colour, price, compare-at price, stock, reserved quantity, barcode, and Active/Inactive/OutOfStock status. Check constraints prevent negative reserved quantity and reserved quantity above stock.
- `inventory_movements`, linked to sellers, products, and product variants, storing seller-visible stock ledger history for single adjustments, bulk imports, checkout reservation creation/release/expiry/confirmation, payment-failed reservation release, return requests, refund completion, explicit seller restock decisions, and admin-applied historical backfill rows with stock/reserved before/after values, quantity delta, related cart/order/reservation/payment/return/refund ids, status before/after, source, reason, actor user id, optional batch reference, and occurrence timestamp.
- `product_images`, linked to products, with URL, storage key, optional media asset id, alt text, sort order, primary flag, and created timestamp. Image binary data is not stored in PostgreSQL.
- `product_listing_revisions`, linked to published products and sellers, staging proposed listing fields, category, brand, descriptions, SEO/merchandising/care/disclaimer presentation fields, tags JSON, attributes JSON, review status, rejection reason, and review timestamps before admin approval applies changes to the live product.
- `product_listing_revision_images`, linked to product listing revisions, storing the proposed final image set with URL, storage key, optional media asset id, alt text, sort order, primary flag, and optional source product image id.
- `product_variant_revisions`, linked to published products and sellers, staging moderation-aware SKU, size, colour, price, compare-at price, barcode, new-variant, and variant-deactivation changes before admin approval applies them to live variants. A filtered unique index allows only one active Draft/PendingReview/Rejected revision per product.
- `product_variant_revision_items`, linked to product variant revisions and optionally source product variants, storing Add/Update/Deactivate operations plus proposed variant fields and initial stock for new variants.
- `media_assets`, storing metadata for API-managed product media originals: provider, bucket, storage key, public URL, seller/product/revision context, original filename, content type, byte size, SHA-256 hash, dimensions, scan status, lifecycle status, timestamps, and cleanup errors.
- `media_asset_variants`, linked to media assets, storing generated `thumb`, `card`, and `detail` WebP variant object metadata.
- `product_attribute_values`, linked to products, storing category-specific attribute values as JSONB.
- `product_reviews`, linked to buyers, sellers, products, orders, and order items, storing verified-buyer review ratings, optional title/body, moderation status, moderation reason/actor/timestamp metadata, and audit timestamps. A unique index enforces one review per order item.
- `ai_product_suggestions`, linked to seller/product drafts, storing AI listing suggestions, prompt/model metadata, quality score, status, JSONB suggested attributes/tags/missing fields/risk flags, and accepted/applied timestamps.
- `ai_suggestion_field_audits`, linked to AI suggestions, storing AI vs seller-final field values and accepted/edited flags.
- `ai_usage_logs`, recording AI feature usage metadata, token/cost estimates, latency, success/failure, and non-secret error details.
- `ai_prompt_versions`, storing prompt templates and version metadata for AI features. Provider API keys or secrets are not stored here.
- `ai_moderation_results`, linked to seller/product listings, storing business-rule moderation risk level, admin-review flag, reason, detected terms, missing fields, and flags as JSONB.
- `product_embeddings`, linked to products, storing AI semantic-search source text, model name, creation timestamp, and a PostgreSQL `vector(1536)` embedding.
- `carts`, linked to buyer profiles and optionally one seller, enforcing one active cart per buyer for the MVP and keeping checked-out carts as historical order sources after order creation.
- `cart_items`, linked to carts, products, and product variants, capturing quantity, unit price, product title, SKU, size, and colour at add time.
- `inventory_reservations`, linked to product variants, buyer profiles, and carts, storing checkout inventory holds with Active/Confirmed/Expired/Cancelled status and expiry timestamps. Reservation services use conditional database updates for PostgreSQL stock holds/releases.
- `orders`, linked to buyer profiles, seller profiles, and source carts, storing checkout order headers with PendingPayment/Paid/Processing/ReadyToShip/Shipped/Delivered/ReturnRequested/Refunded/Cancelled/Disputed/Completed statuses plus server-computed shipping, platform-fee, discount, nullable delivery-address/instruction/verification, delivery-method/rate, pickup-point, and seller-policy snapshot columns for legacy rows, and audit timestamps.
- `order_items`, linked to orders, products, and product variants, snapshotting product title, SKU, size, colour, unit price, and quantity at order creation time.
- `order_status_history`, linked to orders, recording every status transition including the initial PendingPayment creation entry.
- `shipments`, linked to orders, sellers, and buyers, storing manual fulfilment status, carrier/tracking details, ready/in-transit/delivered/failed/returned-to-sender state, provider-neutral carrier booking metadata, provider shipment reference/status/label/error data, package dimensions/weight, sync timestamps, shipped/delivered timestamps, and audit timestamps.
- `shipment_events`, linked to shipments, recording manual fulfilment, tracking, and exception events with status snapshots.
- `return_requests`, linked to orders, buyers, and sellers, storing return status, reason, seller response, buyer dispute metadata, and request/audit timestamps.
- `return_items`, linked to return requests and source order items, storing returned quantity, reason, opened/unsealed flag, and item notes.
- `return_messages`, linked to return requests, storing buyer/seller return workflow messages.
- `return_restock_decisions`, linked to sellers, return requests/items, products, and variants, storing one seller inspection/restock decision per return item with quantity restocked, condition, reason, actor user id, and creation timestamp. Positive restock quantities increment live variant stock and write `ReturnRestocked` inventory movements; zero-restock decisions record inspection context only.
- `payments`, linked to orders and buyer profiles, storing local payment records, configured provider name/reference, retryable checkout URL, amount, currency, status, and paid/failed timestamps. A filtered unique index allows only one active non-failed/non-cancelled payment per order.
- `payment_events`, linked optionally to payments, storing sanitized provider webhook JSON, provider event id, event type, received/processed timestamps, raw-payload redaction timestamp, processing status, and error details. JSON webhooks and form-encoded PayFast ITNs are normalized into JSON before persistence, with signatures/tokens/secrets/card fields redacted. `(Provider, ProviderEventId)` is unique for webhook idempotency. The worker redacts retained raw webhook payload JSON after the configured retention window while preserving event metadata for reconciliation.
- `payment_reconciliation_reviews`, linked to payments, storing finance review evidence from provider dashboards: provider/reference, observed provider status, observed amount/currency, review outcome, reason, reviewer user id, reviewed timestamp, and optional next-review timestamp. These records never mutate payment, order, ledger, cart, reservation, refund, or payout state.
- `refunds`, linked to orders, payments, buyers, sellers, and optionally return requests, storing requested/approved/processing/refunded/failed/rejected status, amount, provider refund reference, reason, requester/approver actor details for dual control, audit timestamps, and a concurrency token for approval processing. Buyer refund read APIs expose only buyer-safe projections and never return provider payloads, finance notes, actor ids, or audit logs.
- `refund_events`, linked to refunds, recording refund request, approval, processing, completion, and failure events.
- `disputes`, linked to orders, buyers, sellers, and optionally return requests, storing dispute status, opening reason, resolution reason, and audit timestamps.
- `dispute_messages`, linked to disputes, storing buyer/seller dispute conversation messages.
- `dispute_evidence`, linked to disputes, storing text/file/image evidence references and descriptions. File binary data is not stored in PostgreSQL.
- `support_tickets`, linked optionally to buyer profiles, seller profiles, orders, products, sellers, and payments, storing support category, status, priority, subject, description, assignment, escalation context, and lifecycle timestamps.
- `support_messages`, linked to support tickets, storing buyer/seller/support/admin messages plus an `is_internal` flag for private support notes that are hidden from buyers and sellers.
- `admin_queue_saved_views`, used by moderation queues and support operations, stores per-admin/support-user saved filters including queue, view, status, category, search, assignment, priority, SLA, sort, and page size.
- Support quality reporting is computed read-only from `support_tickets` and `support_messages` using configurable reporting targets. Phase 10W adds no table because SLA outcomes, first-response timing, resolution timing, category trends, assignee workload, and CSV evidence are derived from existing support records.
- `notifications`, linked to Identity users as recipients, storing notification type, title, message, related entity metadata, read timestamp, in-app visibility flag, and creation timestamp. Buyer and seller notification creation checks category preferences for known transactional notification types. Buyer/seller real-time SignalR delivery uses these persisted rows as the source of truth and does not add storage tables.
- `notification_email_deliveries`, linked to notifications, storing durable transactional email outbox records with recipient email, subject/body, delivery status, attempts, retry timestamp, sent timestamp, failure reason, and audit timestamps.
- `ad_campaigns`, linked to sellers, storing seller ad campaign type, status, run dates, review lifecycle timestamps, and rejection reason.
- `ad_campaign_products`, linked to ad campaigns and promoted products.
- `ad_budgets`, linked one-to-one to ad campaigns, storing daily budget, total budget, max CPC, spent amount, and currency.
- `ad_impressions`, `ad_clicks`, `ad_conversions`, and `ad_charges`, storing ad event/reporting data captured by public tracking endpoints and backend paid-order attribution.
- `seller_ad_credits`, storing seller advertising credit balances per currency.
- `ledger_entries`, append-only internal ledger rows linked optionally to orders, order items, sellers, buyers, and payments, with type, amount, currency, direction, description, and creation timestamp.
- `seller_balances`, storing seller pending, available, and held balances per currency.
- `commission_rules`, storing commission and provider-fee rule metadata for future admin configuration. Current ledger processing uses configured default fee options.
- `seller_payouts`, linked to sellers, storing net payout amount, currency, Pending/OnHold/Available/Processing/PaidOut/Reversed/Failed status, hold/release/availability/processing/provider audit details, provider references/status, terminal timestamps, and a concurrency token.
- `seller_payout_items`, linked to seller payouts and source ledger entries, tying payout records back to order/payment ledger activity and tracking refund-adjusted amounts.
- `seller_payout_adjustments`, linked to refunds, seller payouts, payout items where applicable, and refund ledger entries, recording payout reductions or recovery-required signals without mutating paid-out provider records.

Product image files are stored outside PostgreSQL. The local development provider writes files under the configured ignored media root and serves them through `/media/product-images/...`; the S3-compatible provider stores originals and generated WebP variants in object storage and returns CDN/public URLs. Media metadata, scanner status, lifecycle status, and cleanup state are stored in PostgreSQL.

Public product search currently reads from PostgreSQL tables directly when no external/local search index data is available. Prompt 32 added a search indexing abstraction and local in-memory placeholder; it did not add a database table or external search dependency.

Prompt 33 adds pgvector support through the PostgreSQL `vector` extension. Local PostgreSQL must have pgvector installed before applying the `ProductEmbeddingsPgvector` migration; the repository Docker Compose file uses the `pgvector/pgvector:pg16` image for that purpose.

## Rules

- Use relational modeling for financial, order, inventory, and payout data.
- Use JSONB only where flexible category attributes require it.
- Keep payment and ledger records append-oriented.
- Sellers cannot become `Verified` until required profile, storefront, address, and admin-approved payout placeholder data are complete.
- Do not store full sensitive bank data unless it is encrypted/tokenized; current payout storage is placeholder/provider-reference only.
- Add migrations through the Infrastructure project.
- Sensitive admin workflows must write `audit_logs` through the shared audit logging service.
